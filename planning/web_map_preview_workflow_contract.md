# Web Map Preview Workflow Contract

## Purpose

Specify the browser map preview workflow before choosing a map library, tile server, raster processing stack, or hosted deployment architecture.

This completes P9.3. The main constraint is that the recovered catalog has preview signals but does not yet provide enough validated spatial metadata to safely promise browser map rendering for every layer.

## Current Archive Signals

Recovered catalog evidence as of P9.3:

- total recovered catalog records: 2,183;
- data-layer records: 418;
- virtual-layer records: 1,765;
- all records have at least one TIFF payload member recorded in `raster_member_paths`;
- all data-layer records have WMS XML sidecar members recorded in `wms_member_paths`;
- virtual-layer records do not currently have WMS XML sidecar members;
- compact recovered metadata currently has no populated authoritative `crs` values;
- compact recovered metadata currently has no populated authoritative `spatial_extent` values.

Implication: map preview should be planned as a validation and preprocessing workflow, not as an immediate assumption that every recovered TIFF can be rendered correctly in a browser.

## Preview Eligibility Classes

The browser should distinguish preview status from mere payload existence.

Eligibility classes:

- `not_evaluated`: catalog metadata says a raster exists, but no preview validation has run;
- `metadata_candidate`: recovered metadata includes raster or WMS sidecar signals, but CRS/extent/georeferencing are not validated;
- `preview_ready`: a derived preview artifact exists and has validated CRS, extent, and rendering metadata;
- `preview_unavailable`: the layer cannot be previewed yet because required spatial metadata, payload access, or processing support is missing;
- `preview_error`: preview generation was attempted and failed with a recorded, non-secret-safe diagnostic message.

The catalog browser should expose a preview hint, but the map view should only render layers marked `preview_ready`.

## First Preview Artifact Strategy

The first implementation should prefer precomputed preview artifacts over dynamic raw-raster rendering in the browser.

Recommended progression:

1. Generate static thumbnail or low-resolution quicklook artifacts for representative records.
2. Generate georeferenced preview metadata after CRS and extent are validated.
3. Generate map tiles or Cloud Optimized GeoTIFF derivatives only after the validation workflow proves stable.
4. Consider dynamic server-side rendering only after hosting, caching, and cost boundaries are explicit.

Reasons:

- raw TIFF payloads live in DataLad/object storage, not in the browser bundle;
- browser page load must not trigger DataLad retrieval;
- CRS and extent are not yet authoritative in compact metadata;
- hosted dynamic rendering could create significant CPU, storage, and egress cost.

## Preview Artifact Metadata

Each previewable layer should eventually have a small tracked or service-generated preview metadata record.

Minimum fields:

- `dataset_id`;
- `preview_status`;
- `preview_kind`: thumbnail, tilejson, cog, static_image, or none;
- `source_payload_path`;
- `derived_artifact_path` or URL;
- `crs`;
- `bounds`;
- `bounds_crs`;
- `pixel_size` or approximate resolution when known;
- `generated_at`;
- `generator`;
- `validation_status`;
- `failure_reason` when unavailable or failed.

This metadata should be generated by a reproducible workflow and should not be hand-authored for broad batches.

## CRS And Extent Requirements

Before any layer is shown on a zoomable map, the project must validate:

- source TIFF georeferencing can be read;
- CRS is known and transformable to the browser map CRS;
- bounds are known;
- bounds are plausible for British Columbia or the intended layer domain;
- nodata handling is understood enough to avoid misleading all-filled previews;
- categorical versus continuous raster behavior is identified where styling depends on it.

If these checks have not passed, the UI should show a preview-unavailable state with the reason.

## Map UI Responsibilities

The first map preview view should:

- be reachable from a dataset detail page;
- show dataset title, ID, and preview status;
- show one preview layer at a time unless multi-layer preview is explicitly added later;
- show basic zoom/pan controls only when a georeferenced preview artifact exists;
- show legend or value notes only when recovered metadata supports it;
- preserve a link back to the dataset detail page;
- expose source/provenance notes near the preview status.

The first map preview view should not:

- trigger raw payload retrieval from the browser;
- expose S3, DataLad, git-annex, Arbutus, or Chinook credentials;
- promise AOI clipping or download generation;
- infer missing CRS/extent silently;
- render unknown data as if it were authoritative.

## Route Shape

Stable route shape:

```text
/catalog/{dataset_id}/preview
```

The route should be shareable. For unavailable previews, the route should still render a useful status page explaining why preview is unavailable and what evidence exists.

## Service/API Boundary

The map preview layer may use:

- `HectaresBC().get(dataset_id)` for catalog lookup;
- package path-resolution semantics for source payload paths;
- a future preview metadata index for preview status and derived artifact location;
- server-side workers or scripts for preview generation only in explicit operational workflows.

The browser should not parse raw ZIP/TIFF payloads directly in the first implementation.

If a static preview index is used, it should be generated by a tracked script and validated against the package catalog.

## Failure States

Required user-facing states:

- dataset not found;
- no raster payload signal;
- raster exists but preview validation has not run;
- CRS unknown;
- extent unknown;
- derived preview missing;
- preview generation failed;
- preview service unavailable.

Messages should avoid private local paths, secrets, stack traces, and credential hints.

## Testing Boundary

Default tests and smoke checks should not require:

- Arbutus or Chinook credentials;
- UBC CWL;
- DataLad network retrieval;
- raw TIFF or ZIP payload content;
- a tile server;
- a hosted worker queue.

Routine verification can test:

- preview eligibility classification from compact catalog fields;
- route behavior for representative data-layer and virtual-layer records;
- unavailable-state rendering for records without validated preview metadata;
- static preview-index schema validation if introduced.

Operational validation over real payloads belongs in a separate preview-generation phase.

## Open Questions For P9.4 And P9.6

Questions for P9.4:

- Should AOI drawing be allowed before a preview artifact exists?
- Should AOI selection operate against a standard grid even when layer preview is unavailable?
- How should AOI bounds be validated if layer-specific extents remain unavailable?

Questions for P9.6:

- Where should derived preview artifacts live?
- Should preview tiles be public static objects or served through an authenticated service?
- What cache/lifecycle rules should apply to generated preview artifacts?
- What worker resources are acceptable for preview generation?

## Implementation Deferrals

P9.3 does not implement:

- a map component;
- tile generation;
- raster reprojection;
- thumbnail generation;
- object-store publication of preview artifacts;
- a preview metadata schema file;
- frontend framework selection.

Those belong to a later implementation phase after the workflow and validation requirements are accepted.
