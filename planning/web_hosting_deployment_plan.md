# Web Hosting And Deployment Plan

## Purpose

Define a practical hosting and deployment plan for the future browser app without implementing infrastructure in Phase 9.

This completes P9.6. The plan separates low-cost public discovery from controlled processing workflows that need authentication, credentials, workers, queues, and output lifecycle management.

## Deployment Tiers

The browser app should be designed around three tiers.

Tier 1: static discovery

- catalog search and metadata browsing;
- static generated catalog JSON or a read-only package-backed API;
- public or low-friction access;
- no raw payload retrieval;
- no credentials;
- no hosted processing.

Tier 2: derived preview artifacts

- static thumbnails, quicklooks, tiles, or preview metadata generated by controlled workflows;
- public or access-controlled read access depending on cost and sensitivity;
- no browser access to raw payload credentials;
- preview artifact lifecycle and regeneration documented.

Tier 3: controlled processing

- AOI validation that depends on layer-specific processing metadata;
- download request submission;
- queue and worker execution;
- temporary output storage;
- authentication required;
- rate limits, size limits, and operator diagnostics required.

The initial implementation should start with Tier 1. Tier 2 and Tier 3 require additional implementation and operational validation.

## Recommended Initial Hosting Posture

Recommended first hosted posture:

- static or near-static catalog browser;
- no public job creation;
- no public raw-payload retrieval;
- no browser-side credentials;
- package/CLI command snippets for users who need local retrieval;
- clear unavailable states for preview, AOI processing, and custom downloads until their services exist.

This gets the recovered catalog into a browser without exposing storage credentials or accepting cost-generating work.

## Authentication Boundary

Access-control should be split by workflow.

Public or low-friction:

- catalog search;
- dataset detail pages;
- provenance and uncertainty notes;
- static documentation;
- package/CLI examples.

Authenticated:

- AOI draft persistence, if server-side drafts are used;
- download request submission;
- request status pages containing user-specific AOI or outputs;
- output collection;
- operator diagnostics.

The first hosted job-creating deployment may use UBC CWL or another access-control layer. The implementation should keep authentication isolated enough that a self-hosted deployment can use a different provider.

## Credential Handling

Credentials must remain server-side or user-local.

Rules:

- do not bundle object-store credentials into browser assets;
- do not expose DataLad/git-annex remote secrets to client code;
- do not commit Arbutus, Chinook, S3, CWL, or service account secrets;
- use environment variables or user-local config for service credentials;
- keep the existing user-local `~/.config/fresh-hectaresbc/` pattern for local operational workflows;
- redact credential-like values from logs and user-facing errors.

Any worker that retrieves raw payloads must run in a controlled server-side environment.

## Request Persistence

Controlled processing requires a request store.

Minimum request-store responsibilities:

- request ID generation;
- authenticated owner or submitter metadata;
- request payload and AOI intent;
- validation state;
- job state;
- user-safe status messages;
- timestamps for creation, update, completion, and expiration;
- output references;
- failure summaries.

Implementation can start with a simple database when job creation is introduced. Static catalog hosting does not require request persistence.

## Output Storage And Lifecycle

Prepared outputs should be temporary.

Required policies:

- storage backend;
- output key/path naming convention;
- maximum output size;
- retention period;
- cleanup mechanism;
- authenticated download route or temporary signed URL strategy;
- output manifest format;
- operator recovery path for failed cleanup.

Default posture:

- expire prepared outputs automatically;
- retain request metadata only as long as needed for audit/debugging;
- avoid storing user AOI geometry longer than necessary;
- do not treat hosted outputs as canonical data products unless a later publication workflow explicitly promotes them.

## Queue And Worker Responsibilities

Processing should run outside the browser process.

Worker responsibilities:

- retrieve needed source payloads server-side;
- validate CRS, extent, and AOI compatibility;
- run clipping or tile/bundle generation;
- write outputs to controlled storage;
- update request state;
- emit user-safe failure summaries.

Web/service responsibilities:

- authenticate users;
- validate request shape and limits;
- enqueue work;
- show status;
- provide output collection routes.

Browser responsibilities:

- collect user intent;
- display validation/status;
- never retrieve raw payloads or handle credentials directly.

## Observability And Diagnostics

Hosted workflows need operator diagnostics before public exposure.

Minimum observability:

- request state transition logs;
- worker start/finish/failure logs;
- output cleanup logs;
- rate-limit and validation rejection counts;
- storage usage summaries;
- queue depth and worker health;
- user-safe error IDs that can be correlated with operator logs.

Diagnostics shown to users must avoid stack traces, secrets, internal host paths, and credential configuration.

## Cost And Abuse Controls

Controls required before enabling job creation:

- authentication;
- per-user submission rate limits;
- concurrent job limits;
- maximum AOI area;
- maximum vertex count;
- maximum tile count;
- maximum selected dataset count;
- maximum estimated output size;
- maximum runtime;
- output retention limits;
- administrative disable switch for job submission.

Unauthenticated traffic should be limited to low-cost discovery surfaces.

## Self-Hosting Expectations

The project may support self-hosting, but self-hosting should not require hidden maintainer knowledge.

Future self-hosting documentation should define:

- required Python package version;
- static catalog generation command, if used;
- object-store configuration variables;
- request-store backend;
- worker command;
- authentication provider configuration;
- output storage configuration;
- cleanup command or lifecycle rule;
- smoke-test commands.

Self-hosters should be able to run catalog-only mode without object-store credentials or workers.

## Default Validation

Default development and CI-style checks should not require:

- UBC CWL;
- Arbutus or Chinook credentials;
- object-store writes;
- DataLad network retrieval;
- raw TIFF/ZIP payload content;
- hosted workers;
- production queues.

Routine validation can cover:

- static catalog generation or package-backed catalog loading;
- route and state behavior;
- fake request persistence;
- fake queue/worker state transitions;
- user-safe error rendering;
- configuration validation with dummy values.

Operational validation should separately cover:

- authenticated hosted request submission;
- server-side credential loading;
- DataLad/object-store retrieval;
- real preview generation;
- real clipping/output generation;
- output storage lifecycle;
- cleanup behavior.

## Phase 9 Closeout Decision

Phase 9 specifies the browser app workflow and deployment boundaries. It does not implement the browser app.

Recommended next implementation phase should start with a narrow catalog-only browser surface and avoid job creation until the catalog UI, static/service boundary, and deployment choice are proven.

## Implementation Deferrals

P9.6 does not implement:

- frontend app scaffold;
- authentication integration;
- HTTP service;
- database;
- worker;
- queue;
- object-store output writes;
- preview generation;
- clipping;
- production deployment.

Those belong to later implementation phases.
